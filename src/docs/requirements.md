# Wi-Fi／URL QRコード生成 Webアプリ 要件定義

## 1. 目的

- 来訪者やスタッフが Wi-Fi に迅速に接続できるよう、SSID／パスワードから接続用 QR コードを生成し共有する。
- URL へ誘導するための QR コードも同一画面で生成し、印刷物やサイネージ用に活用できるようにする。
- 情報漏洩リスクを排除するため、QR 生成処理をブラウザのみで完結させ、バックエンド通信を発生させない。

## 2. スコープ

- クライアント: SPA（想定は HTML/CSS/JavaScript または既存フロントフレームワーク）
- バックエンド: Go サーバーは静的ファイル配信のみ（API なし）
- 対象ブラウザ: 最新の Chrome／Edge／Safari（サポートブラウザは別途 QA で確定）

## 3. 機能要件

### 3.1 画面構成

1. **モードトグル**
   - 画面上部に Wi-Fi / URL を切り替えるトグルスイッチ。
   - 初期選択は Wi-Fi モード。
2. **Wi-Fi モード**
   - 入力フォーム 2 つ: `SSID`, `パスワード`。
   - 必須／任意設定: SSID 必須、パスワードは空でも可（オープンネット想定）。
   - フォーム入力に応じて QR コードを即時プレビュー（リアルタイム更新 or 「生成」ボタン押下後でも可、UX 観点で後者推奨）。
3. **URL モード**
   - 入力フォーム 1 つ: `URL`。
   - URL バリデーション（プロトコル付き形式推奨。http/https が無い場合は https を自動付与する仕様も検討）。
4. **QR コード表示エリア**
   - 生成された QR コードのプレビュー画像を表示。
   - Wi-Fi モードでは `WIFI:T:<encryption>;S:<SSID>;P:<password>;H:false;;` 形式でエンコード。
   - URL モードでは入力 URL をそのままエンコード。
5. **ダウンロード操作**
   - 「PNG ダウンロード」ボタンを表示し、生成済み QR を画像として保存できる。
   - 出力形式は PNG のみ。ファイル名は `wifi-qr-YYYYMMDD.png` や `url-qr-YYYYMMDD.png` 等で自動付与。

### 3.2 バリデーション／エラーハンドリング

- 未入力や形式不備時は QR を生成せず、フォーム下にエラーを表示。
- Wi-Fi モード: SSID が空ならエラー。パスワードは 0〜63 文字に制限。
- URL モード: URL 形式を簡易チェックし、不正な場合はエラーメッセージ。

### 3.3 生成仕様

- QR コードライブラリ例: `qr-code-styling`, `qrcode.js`, `qrious` など純クライアント実装可能な OSS。
- エンコード/画像生成はすべてブラウザ上で実行。バックエンド API には送信しない。
- 生成結果は `<canvas>` もしくは `<img>` 要素に描画し、`toDataURL` でダウンロードデータを取得。

### 3.4 UX 補助要件

- 入力補助: クリップボード貼り付けボタン、例示プレースホルダ（例: SSID: `OfficeWiFi`, URL: `https://example.com`）。
- QR 表示領域にガイドテキスト（「コードをスキャンして接続」など）を任意で表示できる拡張余地。
- カラースキームは白／黒のみを使用し、ブランドロゴは表示しない。
- 画面要素は QR 生成に必要な最低限（トグル、フォーム、QR プレビュー、ダウンロードボタン、注意書き程度）のみに絞り、余計な情報や装飾を排除してフラット・スタイリッシュな UI を目指す。

## 4. 非機能要件

- **セキュリティ**: 入力情報はローカルブラウザのみで保持し、永続化しない（LocalStorage 保存機能は初期実装では持たない）。
- **パフォーマンス**: 一般的なノート PC／スマホで 100ms 以内に QR 生成完了。
- **アクセシビリティ**: フォーム要素にはラベルを関連付け、キーボード操作のみでも利用可能にする。
- **レスポンシブ**: スマホ幅（375px）〜 PC 幅で UI が崩れないようにレイアウトを調整。
- **多言語対応**: i18n ライブラリ（例: i18next）を用いて日本語／英語の切り替えを実装。

## 5. 画面遷移／状態

- 画面は単一ページ。トグル切り替えでフォームとバリデーションロジックが切り替わるのみ。
- ブラウザリロード時に状態は初期化（必要であれば後続対応で入力値保持を検討）。

## 6. 技術方針

- Go バックエンドは静的ファイル提供のみ。QR ロジックはすべてフロントエンドで完結させる。
- クライアント実装は TypeScript + 軽量な構成（例: Vite + プレーン React/Preact or バニラ TS）を採用し、依存ライブラリは最小限に保つ。
- 将来的に API 化する場合でも、デフォルトはオフライン動作を優先。
- QR 表現のテストはユニット（入力→エンコード文字列）と E2E（ブラウザでのダウンロード操作）で担保。
- i18n ライブラリは `i18next` を採用。シンプルな JSON リソースで日英文言を管理し、トグルまたはブラウザ設定に基づいて動的に切り替える。
- UI／ビルド構成は軽量さと開発容易性を優先し、必要最小限のビルドステップで済むようにする。

## 7. リスクと対応

- **入力ミス**: 明確なエラーメッセージとリアルタイム検証で低減。
- **URL フィルタリング**: 不正 URL への誘導防止は別施策（ホワイトリスト等）として検討。
- **印刷品質**: 300px 以上の PNG を標準出力、必要に応じてオプションでサイズ変更。

## 8. 残課題・確認事項

1. 現時点で追加確認事項なし。ブランドカラー（白黒のみ）、PNG 固定ダウンロード、i18n 必須、ロゴ非表示は本要件で確定。

## 10. 回答: バックエンド通信なしは可能か？

- 可能。QR コード生成用ライブラリをフロントエンドに組み込み、入力データをブラウザ内でエンコード・画像化することで、Go バックエンドとの通信を行わずに要件を満たせる。
- Go サーバーは静的ファイル（HTML/CSS/JS）を配信する役割に限定し、ネットワーク情報はクライアント外へ送信しない構成とする。

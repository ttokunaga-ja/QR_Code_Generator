### 2025-11-30 実装フェーズ: Docker ビルド失敗（npm ci）の根本対応

**作業内容**:
- Docker のフロントエンドビルドステージで `npm ci` が外部ネットワークへ到達できず `ECONNRESET` が発生する問題を調査。社内プロキシや独自レジストリ環境で `npm` の設定が渡せていないことが原因。

- `docker-compose.yml` でも同じ build args を受け取れるようにし、`.env` などから簡単に設定できるよう調整。
- README の Docker セクションへ、プロキシや独自レジストリを指定する方法を追記。

**判断事項・変更点**:
- `npm ci` 自体の挙動は変えず、ネットワーク設定を外部から与えられるようにすることで根本原因（ビルド環境での接続設定不足）を解消。

### 2025-11-30 実装フェーズ: フォーム入力検知／QR プレビューが表示されない不具合

**原因**:
- `hasQR` が false の間は `<canvas>` 自体を描画していなかったため、`generateWiFiQR` / `generateURLQR` が実行されても `canvasRef` が常に `null` となり QR が描画されなかった。
- その結果、フォーム入力をしても `hasQR` が true に切り替わらず、プレビューが表示されない状態になっていた。

**対策**:
- QR プレビュー枠内に `<canvas>` を常時レンダリングしつつ、`hasQR` が false のときは `opacity-0` にして非表示にする方式へ変更。プレースホルダーは `!hasQR` の間だけ描画するため、視覚的には「どちらか一方」しか表示されない。
- `npm run build` でビルド成功を確認。

**次のアクション**:
- 入力→QR 生成 → ダウンロードまでのフローを手動確認し、追加の見た目調整が必要か判断する。

### 2025-11-30 実装フェーズ: 入力状態ベースでのプレビュー制御

**作業内容**:
- 要件に合わせ、QR プレビューとプレースホルダーの表示切替を `hasQR` 状態ではなくフォーム入力の有無で判断するようリファクタリング。
- Wi-Fi モードでは SSID が入力されているか、URL モードでは URL が入力されエラーが無いかで `shouldShowQR` を算出。これに応じて `<canvas>` の表示／非表示とダウンロードボタンの活性状態を制御。
- `<canvas>` は常時 DOM に残しつつ `opacity` 切り替えのみ行い、テキストと同時に表示されることがないようにした。

# Wi-Fi/URL QRコード生成アプリ 要件理解・デザイン方針メモ

**作成日**: 2025-11-30  
**参照ドキュメント**: `docs/requirements.md`, `docs/figmaai-task.md`, `docs/figmaai-prompts.md`

---

## 1. 要件理解の確認

### 1.1 プロジェクト概要
- **目的**: Wi-Fi 接続用およびURL誘導用のQRコードをブラウザで生成し、セキュアに共有する
- **最重要方針**: 情報漏洩リスク排除のため、すべての処理をブラウザ内で完結させる
- **バックエンド通信**: 一切行わない（Go サーバーは静的ファイル配信のみ）

### 1.2 技術構成
- **バックエンド**: Go（静的ファイル配信専用、API なし）
- **デプロイ**: Docker コンテナ内で Go サーバーとフロント資産が同居
- **フロントエンド**:
  - QR 生成: 純クライアントライブラリ（qrcode.js, qr-code-styling 等）
  - バリデーション: ブラウザ内 JavaScript
  - 多言語対応: i18next（JSON リソースで日英管理）
  - 画像出力: Canvas API で PNG 生成・ダウンロード

### 1.3 機能スコープ
**必須要素**:
1. モード切替トグル（Wi-Fi / URL）
2. Wi-Fi モード: SSID（必須）+ パスワード（任意）入力
3. URL モード: URL 入力（https:// 自動補完）
4. QR コードプレビュー領域（300px 以上）
5. PNG ダウンロードボタン
6. 入力注意書き・エラーメッセージ

**除外要素**:
- ブランドロゴ
- 余計な装飾や情報
- バックエンド API との通信
- 永続化機能（初期実装では LocalStorage も使わない）

## 2. デザインアプローチ

### 2.1 デザイン哲学
**ミニマリズム最優先**:
- 白と黒のみを使用（グレースケールも最小限）
- フラットデザイン、余白を広く取る
- 視線誘導を明瞭に（トグル → フォーム → QR プレビュー → ダウンロードボタン）
- 情報密度を低く保ち、必要な要素のみを配置

### 2.2 カラーパレット
```
背景: #FFFFFF (白)
テキスト・線・QRコード: #000000 (黒)
エラー: #000000（太字または下線で強調）
フォーカス: #000000（ボーダー太さ変更で表現）
```

### 2.3 タイポグラフィ
- システムフォント系列: `system-ui`, `Segoe UI`, `Noto Sans JP`
- 文字サイズはアクセシビリティを考慮（最小 14px）
- コントラスト比 4.5:1 以上を厳守

### 2.4 レイアウト戦略
**レスポンシブ 2 ブレークポイント**:
- **モバイル**: 375px 幅（縦スタック、余白 16px）
- **デスクトップ**: 1440px 幅（中央寄せ、最大幅 600px 程度、余白拡大）

**構成順序**:
```
[画面上部]
├─ モード切替トグル（Wi-Fi / URL）
│
[入力セクション]
├─ Wi-Fi モード: SSID入力 + パスワード入力
│  └─ 注意書き: 「SSID は必須です」
├─ URL モード: URL 入力
│  └─ 注意書き: 「https:// を含めてください」
│
[出力セクション]
├─ QR コードプレビュー（300×300px 以上）
├─ PNG ダウンロードボタン
│
[フッター]
└─ 簡潔な説明文（任意）
```

### 2.5 コンポーネント設計
**再利用可能コンポーネント**:
1. `Toggle/WifiUrl`: 2 状態トグルスイッチ
2. `Input/Text`: 通常/フォーカス/エラー 3 ステート
3. `Button/Primary`: ダウンロード用プライマリボタン
4. `QRPreview`: Canvas 要素を含むプレビュー枠
5. `ErrorText`: エラーメッセージ表示用

**i18n キー対応例**:
```json
{
  "mode.wifi": "Wi-Fi",
  "mode.url": "URL",
  "label.ssid": "SSID",
  "label.password": "Password",
  "label.url": "URL",
  "action.download": "Download PNG",
  "error.ssid_required": "SSID is required",
  "error.invalid_url": "Invalid URL format"
}
```

## 3. Docker/Go 同居前提の技術考慮

### 3.1 ファイル配置
```
Docker コンテナ内構成:
/app
├── main.go              (Go 静的ファイルサーバー)
├── public/              (ビルド済みフロント資産)
│   ├── index.html
│   ├── app.js
│   ├── app.css
│   └── i18n/
│       ├── ja.json
│       └── en.json
└── ...
```

### 3.2 ビルドフロー
1. フロントエンド: TypeScript + Vite でビルド
2. 成果物を `public/` に出力
3. Docker イメージに Go バイナリとともにパッケージング
4. Go サーバーが `http.FileServer` で `public/` を配信

### 3.3 セキュリティ担保
- すべての QR 生成処理をブラウザで実行
- 入力データはサーバーへ送信しない
- Canvas API で PNG 生成後、`toDataURL` または `toBlob` でローカルダウンロード
- i18n リソースも静的 JSON として配信

# FigmaAI プロンプト（フェーズ分解）

下記は FigmaAI に順次入力してフロントエンドデザインを完成させるためのプロンプト例。  
共通前提として、必ず docs/web-app-requirements.md の要件定義を参照し、「Go バックエンド（静的配信）が同一 Docker コンテナ内で稼働し、フロント側は純クライアントで QR 生成・i18n を行う」ことを強調する。  
また、最終的な実装では docs/dev_log.md に作業内容を毎回記録する旨をノートに含め、各フェーズの出力末尾で「次のフェーズ用プロンプトを最新状況に合わせて見直し・修正する」タスクを必ず明記させる。  
各フェーズのプロンプトには期待する出力内容を明記している。

---

## Phase 0: コンテキスト共有
```
前提: docs/web-app-requirements.md の要件定義を必ず読み込み、その内容に完全準拠すること。同一 Docker コンテナ内で Go バックエンドが静的配信のみを担当し、フロントエンドはブラウザ内で QR 生成・バリデーション・i18next による多言語対応を完結させる。
白黒のみのミニマルデザインを採用し、余計な情報やロゴは置かない。画面要素はモード切替トグル、Wi-Fi/URL フォーム、QR プレビュー、PNG ダウンロードボタン、注意書きに限定する。
作業開始前に、要件理解・デザインアプローチ・Docker/Go 同居前提をまとめたテキストノートを作成し、docs/dev_log.md に実装内容を都度記録するように指示するコメントを配置する。
出力: 上記内容を踏まえた確認メモとデザイン方針のアウトライン（日本語）。最後に「Phase 1 プロンプトを最新状況で調整する」と書き添える。
```

## Phase 1: ワイヤーフレーム
```
docs/web-app-requirements.md の要件を参照しつつ、375px（モバイル）と 1440px（デスクトップ）の 2 レイアウトを単一ページで低忠実度ワイヤーフレーム化する。同一 Docker コンテナ内の Go サーバーからフロント資産が配信される点をアノテーションで明示する。
要素: 上部トグル（Wi-Fi/URL）、Wi-Fi モード（SSID・パスワード入力と注意書き）、URL モード（URL 入力と注意書き）、300px 以上の QR プレビュー、PNG ダウンロードボタン。情報量は最小限に保ち、フラットで整然としたレイアウトにする。
ノートに「実装時は docs/dev_log.md にワイヤーフレーム作業の記録を残すこと」を必ず記載。
出力: モバイル／デスクトップ各 1 枚のワイヤーフレーム＋要件準拠メモ＋dev_log 記録指示。最後に「Phase 2 プロンプトを最新状況で調整する」と明記。
```

## Phase 2: コンポーネント定義
```
Phase 1 のワイヤーフレームを基に、再利用可能なコンポーネントを定義する。参照資料は docs/web-app-requirements.md。コンポーネント名は英語（例: Toggle/WifiUrl, Input/Text, Button/Primary）。
対象: トグル（Wi-Fi/URL 2 状態）、入力フィールド（SSID・Password・URL の通常／フォーカス／エラー）、プライマリボタン（Generate, Download PNG）、QR プレビュー枠、注意書きテキスト。各コンポーネントに i18next のキー（例: label.ssid, action.download）を注記する。
ノートで「コンポーネント実装時は docs/dev_log.md に反映内容を記録」と明示する。
出力: コンポーネント一覧、ステート差分、i18n キー指定、dev_log 記録指示を含むボード。最後に「Phase 3 プロンプトを最新状況で調整する」と記載。
```

## Phase 3: ビジュアルデザイン
```
docs/web-app-requirements.md の非機能要件に従い、白黒のみの洗練されたビジュアルを適用する。フォントは system-ui / Segoe UI / Noto Sans JP を想定し、コントラスト比 4.5:1 以上を満たす。背景は余白と細線で構成し、色要素は最小限にする。
JA/EN のサンプル文言を双方表示し、i18next での切替を視覚的に示す。資産は Go バックエンドと同じ Docker コンテナに梱包され、外部 CDN に依存しないことを再度注記する。
ノートに「ビジュアル実装時も docs/dev_log.md へ反映内容を記録すること」を記載。
出力: モバイル＋デスクトップのビジュアルアートボード、スタイルトークン、注記、dev_log 指示。最後に「Phase 4 プロンプトを最新状況で調整する」と書く。
```

## Phase 4: ハンドオフ準備
```
最終成果として、モバイル／デスクトップの完成アートボード、コンポーネントバリアント、スタイルガイドを整理する。各テキスト要素に i18n キーを紐付け、QR キャンバスの推奨サイズ（300px 以上）を明記する。
PNG ダウンロードボタン等のエクスポート指示を記載し、Go バックエンドと同一 Docker コンテナ内で静的配信されること、バックエンド API を呼ばないことを開発ノートに追記する。
ノートに「ハンドオフ内容を実装した際は docs/dev_log.md に経緯を記録すること」を明確に書き、余計な UI 要素がないことを確認するチェックを含める。
出力: ハンドオフ用アートボード、コンポーネント仕様書、エクスポート指示、dev_log 記録指示付きノート。最後に「必要があれば新たなフェーズ用プロンプトを再定義する」と明記。
```
